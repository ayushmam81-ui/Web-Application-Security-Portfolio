## 1. Access Control & Privilege Escalation Fundamentals

* **Core Definitions:**
    * **Authentication:** Verifies user identity (e.g., username/password)[cite: 3].
    * **Session Management:** Maintains identity across HTTP requests via unique tokens/cookies[cite: 3].
    * **Access Control:** Determines if an authorized user is permitted to perform a specific action[cite: 3].
* **Access Control Types:**
    * **Vertical:** Restricts/grants access based on privilege levels (e.g., Admin vs. Regular User)[cite: 3].
    * **Horizontal:** Prevents users of the same privilege level from accessing each other's resources[cite: 3].
    * **Context-Dependent:** Restricts functionality based on the application state or user interaction, such as requiring re-confirmation before destructive actions[cite: 3].
* **Broken Access Control (BAC):** Occurs when applications fail to properly restrict unauthenticated or unauthorized users, leading to unauthorized viewing, modification, or deletion of data[cite: 3].
* **Common Vulnerability Patterns:**
    * **Insecure Defaults/Obscurity:** Relying on secret URLs or `robots.txt` paths instead of robust server-side enforcement[cite: 3].
    * **Client-Side Trust:** Relying on untrusted client-side inputs (e.g., predictable IDs or parameters like `admin=true`) to make authorization decisions[cite: 3].
    * **Multi-Step Process Failures:** Failing to verify permissions at every stage of a process, allowing attackers to skip steps[cite: 3].
* **Remediation:** Implement strict server-side authorization checks at every step, abandon "security by obscurity," and ensure access decisions are never based on client-controlled parameters[cite: 3].

## 2. Cross-Site Request Forgery (CSRF) Analysis

* **Objective:** Audit session-handling mechanics to identify state-changing actions vulnerable to cross-site request hijacking.
* **Vulnerability Identified:** The application relied exclusively on automated, cookie-based sessions for identity validation without integrating unpredictable server-side verification factors.

### Defense & Remediation Framework
* **CSRF Tokens:** Implement unique, secret, and unpredictable tokens generated by the server per user session.
* **SameSite Cookie Attributes:** Configure session cookies with `SameSite=Lax` or `SameSite=Strict`.
* **Referer/Origin Header Validation:** Verify HTTP headers to ensure requests originate from trusted domains.
* **Re-authentication:** Force re-authentication for critical operations.

## 3. Business Logic & Parameter Manipulation

* **Objective:** Evaluate multi-step e-commerce workflows and registration mechanics to ensure financial constraints, access states, and business rules are enforced at the server layer.
* **Vulnerabilities Tested:**
    * **Price Manipulation:** Altering client-side price parameters via Burp Suite.
    * **Boundary Value Injection:** Using negative quantities to offset order totals.
    * **Workflow Rule Stacking:** Bypassing promo code limits.
* **Remediation:** All transaction parameters must be hard-validated and calculated exclusively on the server side using strict whitelists.

## 4. Registration Logic & Access Control Bypass

* **Objective:** Audit role-based access control configurations and entry parameters on administrative panels.
* **Vulnerability Identified:** Inconsistent domain string validation during public profile creation allowed administrative escalation.
* **Remediation:** Enforced server-side domain verification using strict regular expressions and a default 'deny-all' stance for privileged paths.

## 7. Directory Traversal Filter Evasion

* **Objective:** Navigate out of restricted application file structures to test input verification.
* **Exploitation:** Used absolute path references and nested traversal patterns (`....//`) to bypass weak sanitization.
* **Remediation:** Shifted to a hard whitelisting framework for file extensions and directory paths.
